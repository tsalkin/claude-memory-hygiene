# claude-memory-hygiene

> Держит INDEX файловой памяти Claude Code (`MEMORY.md`) в рамках бюджета — **не теряя знаний**. Один файл, только stdlib, MIT.

**Storage is not memory.** Постоянно загружаемый индекс — это рабочая память, и ему нужен бюджет, как любому кэшу. Память, которая никогда не забывает, — это не фича, а утечка, которая в итоге ослепляет агента.

---

## Проблема

Claude Code даёт каждому проекту файловую «вторую память»: множество маленьких `*.md`-файлов-тем плюс один индекс `MEMORY.md`, всё в `~/.claude/projects/<slug>/memory/`. Harness грузит `MEMORY.md` в контекст **каждую сессию** (сами файлы-темы подтягиваются по релевантности по запросу), поэтому у индекса есть жёсткий бюджет (~24.4 KB — за его пределами появляется предупреждение *«index entries too long»*). Со временем его раздувают две вещи: (а) файлы состояния «на сессию» / «на день» — каждый добавляет в индекс **постоянную** строку и накапливается бесконечно; (б) строки индекса разрастаются из хука в абзац. Как только `MEMORY.md` выходит за бюджет, harness грузит лишь **часть** индекса — и агент **слепнет** к части собственной памяти. Знание при этом никуда не делось с диска, оно просто перестаёт находиться в момент recall (тихая потеря знаний).

## Идея фикса в одной фразе

**Чистота INDEX — это не то же самое, что удаление знаний.** Факты живут в файлах-темах; `MEMORY.md` лишь указывает на них. Значит, индекс можно ужать **без потерь**: укоротить хуки (детали остаются в файле), заархивировать старые файлы состояния (переместить, не удалять), запинить долговечные заметки и перестать плодить файлы «на каждый день».

## Установка

Это один скрипт на стандартной библиотеке Python — без зависимостей, без сборки.

```bash
git clone https://github.com/maxi-mts/claude-memory-hygiene.git
cd claude-memory-hygiene
python memory_hygiene.py report
```

По желанию — глобальный алиас, чтобы запускать из любого проекта:

```bash
alias memhy='python /path/to/claude-memory-hygiene/memory_hygiene.py'
```

## Использование

По умолчанию инструмент работает с памятью **текущего** проекта: каталог памяти выводится из CWD по правилу Claude Code (`~/.claude/projects/<slug>/memory`, где `<slug>` — абсолютный путь проекта со слешами `/`, заменёнными на `-`). То есть запускаешь его изнутри любого проекта.

| Команда | Что делает |
|---|---|
| `report` (по умолчанию) | Размер vs бюджет, число указателей, слишком длинные строки, сколько файлов можно заархивировать. **Read-only.** |
| `lint` | Перечисляет строки индекса длиннее `--max-chars` (укоротить хук — файл остаётся на месте). |
| `archive` | Перемещает **старые** volatile-файлы состояния в `archive/` и убирает их строки из индекса. **DRY-RUN по умолчанию**, нужен `--apply`. |

| Флаг | Назначение |
|---|---|
| `--project PATH` | Вывести каталог памяти из пути к проекту (по умолчанию — CWD). |
| `--dir PATH` | Явный каталог памяти (перебивает `--project`/CWD). |
| `--keep N` | Сколько новейших volatile-файлов оставить в индексе (по умолчанию `3`). |
| `--max-chars N` | Лимит длины строки индекса (по умолчанию `220`). |
| `--volatile REGEX` | Какие имена файлов — кандидаты на архивацию. |
| `--pin REGEX` | Какие имена файлов **никогда** не архивируются (по умолчанию префиксы `feedback_`/`reference_`/`user_`). |
| `--apply` | Для `archive`: реально переместить файлы (иначе dry-run). |

Примеры — копируй и вставляй:

```bash
# Память текущего проекта (каталог выведен из CWD):
python memory_hygiene.py report
python memory_hygiene.py lint --max-chars 220
python memory_hygiene.py archive --keep 3            # dry-run, ничего не меняет
python memory_hygiene.py archive --keep 3 --apply    # реально архивирует

# Любой проект / явный каталог:
python memory_hygiene.py report --project /path/to/some/repo
python memory_hygiene.py report --dir ~/.claude/projects/<slug>/memory
```

## Как выбирается, что архивировать

- **Volatile vs pin.** Кандидат на архивацию — только файл, чьё **имя** матчит `--volatile` (по умолчанию: заметки состояния — `session_state`, `session_log`, `daily_*`, имена с датой `YYYY-MM-DD`) и при этом **не** матчит `--pin`. Долговечные заметки (`feedback_`/`reference_`/`user_`) кандидатами не становятся никогда — даже если проскочат volatile-фильтр, их ловит второй guard `--pin`.
- **Keep N.** Volatile-файлы сортируются «новейшие первыми» (по дате в имени, затем по mtime); новейшие `N` остаются в индексе, всё старше — кандидаты на архивацию.
- **Archive, не delete.** Файлы **перемещаются** в `archive/` — они остаются на диске и доступны для `grep`. Harness авто-грузит только `MEMORY.md`, так что архивные файлы просто покидают постоянно загружаемый индекс; знание остаётся целым.

## Безопасность

- **Dry-run по умолчанию.** `archive` ничего не трогает, пока не передан `--apply` — сначала смотришь план, потом выполняешь.
- **Никогда не удаляет.** Только перемещение в `archive/`; ничего не стирается с диска.
- **Пинит долговечное.** Guard `--pin` исключает `feedback_`/`reference_`/`user_` (и что укажешь сам), так что устойчивая память защищена.
- **Без footgun'а с датами.** Даты берутся из имени файла / mtime — никакого штампования frontmatter не требуется, поэтому нет сценария «пустая дата → заархивировать всё».

## Зачем

Постоянно загружаемому индексу нужен бюджет — ровно как любому кэшу. Когда `MEMORY.md` выходит за рамки, harness молча обрезает его, и агент слепнет к части собственной памяти, хотя факты всё ещё лежат на диске. Этот инструмент чинит ровно один артефакт — всегда загружаемый `MEMORY.md` — и делает это без потерь: укорачивает хуки, архивирует старое состояние и пинит долговечное, чтобы индекс не раздувался снова.
